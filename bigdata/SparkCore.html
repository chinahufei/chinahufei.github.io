spark rdd

如果一个RDD有100个partition， 一个partition有10000条数据

  • map 有多少个数据处理多少次, 返回的是单个数据
  • mapPartition 有多少个分区处理多少次
  • flatMap 操作后应该返回0个或多个输出项(所以，func应该返回一个序列化的数据)
  • glom
    将分区里的数据，按分区的数量放到数组里
  • filter 过滤 rdd.filter(x => x > 2 && x % 2 == 0)
  • sample 取样 rdd.sample(true, 5), 取5个数据
  • zip 将2条数据合在一起，以(key, value)形式输出，如果分区数不同，个数不同，会报错
    val rddZip = sc.parallelize(List(“胡飞”, “jike”, “sqoop”))
    val rddZip2 = sc.parallelize(List(10, 20, 30))
    val resZip = rddZip.zip(rddZip2).collect
  • zipWithIndex 拉链后，添加索引
  • union 把两个rdd的元素合并，不会去重，union后的分区是2个分区数之和
    val rdd1 = sc.parallelize(List(1, 2,3 ,4 , 5), 3)
    val rdd2 = sc.parallelize(List(1, 2,3 ,4 , 5), 2)
    val rdd = rdd1.union(rdd2).collect
    rdd.partitions // 结果为5
  • inetersection 交集 求2个rdd的公共部分
  • substract 在左边不在右边
    rdd1.substract(rdd2).collect
  • cartesian 2个表做join没有条件,生产上慎用
  • distinct 去重
    distinct(5) // 去重后将分区数为5个
    distinct(5).mapPartitionsWithIndex((index, partition) => {
    // 此处打印分区数和索引，分区的原理是按照数字本身对分区数取模产生的
    })
  • coalesce 小文件会有用到
    rdd.coalesece(1) // 改变分区数 默认只能变小
    rdd.coalesece(4, true) // 分区数变大需要加参数为true，有shuffle
    遇到shuffle会切分stage
  • repartition 改变分区，会有shuffle, 和coalesece是一家，只是相互调用
  • sortBy 默认降序，升序变-
  • sortByKey
    val rdd5 = sc.parallelize(List((“胡飞”, 40), (“jike”, 20), (“sqoop”, 90)))
    rdd5.sortBy(-.2).collect // 降序
    rdd5.map(x => (x._2, x._1)).sortByKey().map(x => (x._2, x._1)).collect
  • groupByKey 按key分组，value是一个迭代对象
    val rdd6 = sc.parallelize(List((“胡飞”, 40), (“jike”, 20), (“胡飞”, 100), (“sqoop”, 90)))
    rdd6.groupByKey().mapValues(x => x.sum).collect
  • reduceByKey 经典的就是wc求和
    rdd6.reduceByKey(+).collect
    byKey的算子要作用到(K,V)类型上
  • collect
    适用于数据比较小，将数组中所有数据展示处理，是放在内存中。如果需要看，使用take。生产上使用要慎重。
  • first 调用的是take(1)
  • take 取前2个
  • count 根据分隔符算数量
  • top 会做排序，largest。适用于结果集比较小，否则会OOM。排序是字典序，只是第一位，比如41、2、3、5，排序是5、41。
    如果需要升序 rdd.top(2)(Ordering.by(x => -x))
  • takeOrder(2) 返回最小的2个
  • reduce 叠加 rdd.reduce(+)
  • countByKey 是一个action算子，mapValue().reduceByKey(+).collect()。在看数据分布时，查看每个key有多少个。
  • lookup(“key”) 查找key, 结果使用Array装

wc的groupByKey 和 reduceByKey

第一步： 展平 使用逗号分隔，然后map， 输出为 key,1的键值对列表
第二部： reduceByKey统计出结果
第三部： 调转顺序，排序，再调转顺序，输出

通过数据的shuffle量，可以看出groupByKey的shuffle量要比reduceByKey的shuffle量要大(groupByKey是308B, reduceByKey是295B)
groupByKey 是所有数据都参与shuffle 要慎用
reduceByKey 是部分数据参数shuffle，本地map端有一个预聚合操作combiner, 所以到reduce端数据量会少于groupByKey的shuffle量。减少了网络传输的数据量

宽窄依赖

  • 宽依赖
    一个父RDD的partition会被子RDD的分区使用多次
    有shuffle
    遇到类似的reduceByKey groupByKey

  • 窄依赖
    一个父RDD的分区至多被子RDD的某个分区使用一次
    一个父RDD和一个子RDD的分区是唯一映射 map
    多个父RDD和一个子RDD的分区是唯一映射 union
    特殊的jion, 没有shuffle的join

持久化

  • persist， 可以自己传缓存级别
  • cache
    cache的底层还是调用的是persist,cache只有一个默认的缓存级别MEMORY_ONLY,而persist可以根据情况设置其它的缓存级别。
    StorageLevel有12中缓存级别, 需要传5个参数。
    useDisk：使用硬盘（外存）
    useMemory：使用内存
    useOffHeap：使用堆外内存，这是Java虚拟机里面的概念，堆外内存意味着把内存对象分配在Java虚拟机的堆以外的内存，这些内存直接受操作系统管理（而不是虚拟机）。这样做的结果就是能保持一个较小的堆，以减少垃圾收集对应用的影响。
    deserialized：反序列化，其逆过程序列化（Serialization）是java提供的一种机制，将对象表示成一连串的字节；而反序列化就表示将字节恢复为对象的过程。序列化是对象永久化的一种机制，可以将对象及其属性保存起来，并能在反序列化后直接恢复这个对象
    replication：备份数（在多个节点上备份）
  • 如何选择
    Spark的存储级别旨在在内存使用量和CPU效率之间提供不同的权衡。
    我们建议通过以下过程选择一个：如果您的RDD适合默认存储级别（MEMORY_ONLY），请保持这种状态。这是CPU效率最高的选项，允许RDD上的操作尽可能快地运行。如果不是，请尝试使用MEMORY_ONLY_SER并选择一个快速的序列化库，以使对象的空间效率更高，但访问速度仍然相当快。（Java和Scala）除非用于计算数据集的函数很昂贵，否则它们会过滤掉大量数据。否则，重新计算分区可能与从磁盘读取分区一样快。如果要快速恢复故障（例如，如果使用Spark来处理来自Web应用程序的请求），请使用复制的存储级别。所有存储级别都通过重新计算丢失的数据来提供完全的容错能力，但是复制的存储级别使您可以继续在RDD上运行任务，而不必等待重新计算丢失的分区。

广播变量使程序员可以在每台计算机上保留一个只读变量，而不是将其副本与任务一起发送。例如，可以使用它们以有效的方式为每个节点提供大型输入数据集的副本。Spark还尝试使用有效的广播算法分配广播变量，以降低通信成本。火花动作是通过一组阶段执行的，这些阶段由分布式“随机”操作分开。Spark自动广播每个阶段中任务所需的通用数据。在运行每个任务之前，以这种方式广播的数据以序列化形式缓存并反序列化。这意味着仅当跨多个阶段的任务需要相同的数据或以反序列化形式缓存数据非常重要时，显式创建广播变量才有用。广播变量是通过调用SparkContext.broadcast（v）从变量v创建的。广播变量是v的包装，可以通过调用value方法来访问其值。下面的代码显示了这一点：
